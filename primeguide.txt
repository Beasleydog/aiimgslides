# PrimeIntellect GPU Guide

## Running prime CLI
Must run from WSL:
  wsl bash -lc "prime pods list"

## Listing available GPUs
  prime availability list
  prime availability list --gpu-type L4_24GB

Valid gpu-type values: CPU_NODE, A10_24GB, A100_80GB, A100_40GB, A30_24GB,
  A40_48GB, RTX4090_24GB, L4_24GB, L40_48GB, L40S_48GB, A6000_48GB, H100_80GB,
  RTX6000Ada_48GB, RTX8000_48GB, ... (see error message from bad query for full list)

## Creating a pod
  prime pods create --id <ID> --name <name> --disk-size <GB> --image <image> --yes

Supported images (not all work on all providers):
  ubuntu_22_cuda_12          # works on runpod + massedcompute; base Ubuntu + CUDA, no ML stack
  cuda_12_4_pytorch_2_6      # NOT supported on runpod or massedcompute — avoid
  cuda_12_6_pytorch_2_7      # untested on these providers

Provider gotchas:
  - runpod: supports ubuntu_22_cuda_12; does NOT support cuda_12_x_pytorch_x images
  - massedcompute: same — use ubuntu_22_cuda_12, then pip install what you need

After creating with ubuntu_22_cuda_12, pip install:
  pip install transformers trl peft datasets accelerate pillow qwen-vl-utils torchvision "jinja2>=3.1.0"
  (PyTorch + CUDA already present on the image)

## Managing pods
  prime pods list
  prime pods status <pod-id>
  prime pods terminate <pod-id> --yes
  prime pods history               # see terminated pods

## SSH into a pod
  prime pods ssh <pod-id>          # interactive only — no command passthrough

  SSH details (IP + port) are shown in: prime pods status <pod-id>
  To run commands non-interactively:
    ssh -o StrictHostKeyChecking=no -p <PORT> root@<IP> '<command>'
  massedcompute pods use 'ubuntu' user (not root), standard port 22:
    ssh -o StrictHostKeyChecking=no ubuntu@<IP> '<command>'

## CRITICAL: Run long jobs inside a REMOTE tmux on the pod
  If you run training directly in the prime pods ssh shell, it DIES when the
  SSH connection drops (local tmux kill, network blip, closing laptop, etc).

  CORRECT workflow — always do this for training:
    # 1. SSH in
    prime pods ssh <pod-id>

    # 2. On the pod, start a remote tmux and run training inside it
    tmux new-session -d -s training
    tmux send-keys -t training "python3 train.py 2>&1 | tee train.log" Enter

    # 3. Disconnect however you want — training keeps running on the pod
    exit

  To reconnect and check:
    prime pods ssh <pod-id>
    tmux attach -t training        # live view
    # or non-interactively:
    tmux capture-pane -t training -p | tail -20

## Driving prime pods ssh via local tmux (for scripting)
  tmux new-session -d -s <local-name>
  tmux send-keys -t <local-name> 'prime pods ssh <pod-id>' Enter
  sleep 10   # wait for connection
  tmux send-keys -t <local-name> '<remote command>' Enter
  sleep <N>
  tmux capture-pane -t <local-name> -p   # read output
  tmux kill-session -t <local-name>      # disconnect when done

## Double-tmux pattern (ALWAYS use this for this project)
  Keep a persistent local tmux session named "prime" that holds the
  prime pods ssh connection. Never use direct SSH. Drive everything by
  sending keys into the "prime" session, which forwards to the remote
  "training" session on the pod.

  # Check remote training output non-interactively:
  wsl bash -lc "tmux send-keys -t prime 'tmux capture-pane -t training -p | tail -25' Enter && sleep 4 && tmux capture-pane -t prime -p 2>&1 | tail -25"

  # Send a command to the remote training session (e.g. stop training):
  wsl bash -lc "tmux send-keys -t prime 'tmux send-keys -t training C-c' Enter"

  # Start training inside remote tmux (from scratch or after git pull):
  wsl bash -lc "tmux send-keys -t prime 'tmux send-keys -t training \"git pull && python3 train.py 2>&1 | tee train.log\" Enter' Enter"

  # If local "prime" session died or SSH dropped, reconnect:
  wsl bash -lc "tmux new-session -d -s prime 2>/dev/null; tmux send-keys -t prime 'prime pods ssh <pod-id>' Enter"
  # then sleep 10 before checking

  Key rule: NEVER send commands directly to the pod's shell prompt in the
  prime session — always target the remote "training" tmux session with
  tmux send-keys -t training. The pod shell is just a passthrough.

## Cheap GPU options for ML training (as of 2026-05-15)
  Best value for training a 3B VL model quickly:
    a121b4  L40_48GB  x1  US  massedcompute  ~$0.xx/hr   (Ada Lovelace, fast tensor cores)
    ea69b8  A6000_48GB x1  US  massedcompute  ~$0.xx/hr  (Ampere, 48GB, solid)
    0eabc6  A40_48GB  x2  CA  runpod          ~$0.xx/hr  (storage listed as "Low")
    f5f6fe  A100_80GB x1  US  sxm             ~$1.xx/hr  (best perf, costs more)
    e2c2b8  A100_40GB x1  US  lam             ~$1.xx/hr

  Avoid multi-GPU runpod nodes for small models — single A100/L40 is faster
  due to no inter-GPU comms overhead.

## Workflow for this project
  1. Training data (output/) and code are in git — just git clone on the pod
  2. Create pod, wait for ACTIVE status
  3. git clone https://github.com/Beasleydog/aiimgslides.git && cd aiimgslides
  4. pip install deps (see above)
  5. Start remote tmux, run training inside it
  6. Disconnect freely — job survives
  7. Reconnect anytime: prime pods ssh <pod-id> -> tmux attach -t training
